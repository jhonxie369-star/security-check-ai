## 🎯 核心设计理念

### 1. AI理解优先，工具辅助

**理念**：
- 利用AI的代码理解能力，无需Joern等外部工具
- grep/find只用于定位代码，AI负责深度分析
- AI可以理解上下文、业务逻辑、数据流

**体现**：
- 使用grep搜索关键代码位置
- 读取完整上下文（前后50-100行）
- AI分析数据流、验证逻辑、风险判断

### 2. 端侧AI分析

**理念**：
- 利用AI深度理解代码逻辑和业务场景
- 通过上下文分析减少误报
- 无需安装复杂的分析工具

**优势**：
- 快速部署使用
- 零学习成本

### 3. 上下文分析，减少误报

**理念**：
- 不只看关键词，理解完整逻辑
- 区分测试代码、示例代码、真实代码
- 识别验证函数、安全API

**方法**：
- 读取足够的上下文（前后50-100行）
- 分析函数调用链
- 检查是否有验证逻辑

### 4. 业务理解，发现深层漏洞

**理念**：
- 业务逻辑漏洞比代码漏洞更隐蔽
- 需要理解业务场景才能发现
- AI可以理解业务逻辑

**方法**：
- 先理解项目业务类型
- 识别关键业务流程
- 分析业务逻辑完整性

### 5. 分层设计，按需读取

**理念**：
- knowledge：跨语言的安全知识（理论）
- patterns：语言特定的检查模式（方法）
- payloads：漏洞验证模板（实践）
- lessons-learned：经验积累（优化）

**好处**：
- 按需读取，避免一次读太多
- 知识复用，减少重复
- 易于扩展，添加新语言只需加patterns

## 🔄 工作流程

### AI检查流程

```
1. 项目分析
   ↓ (识别语言、框架、业务类型)
2. 读取知识库
   ↓ (理解安全理论)
3. 读取patterns
   ↓ (获取检查方法)
4. grep搜索关键代码
   ↓ (定位可疑位置)
5. 读取完整上下文
   ↓ (理解代码逻辑)
6. AI深度分析
   ↓ (判断真实风险)
7. 生成报告并保存
   ↓
8. 询问用户是否继续
```

### 信息流动

```
knowledge (理论)
    ↓
patterns (方法) + 项目代码 (实际)
    ↓
grep搜索 (定位)
    ↓
读取上下文 (理解)
    ↓
AI分析 (判断)
    ↓
报告 (输出)
```

## 🎓 设计哲学

### 核心思想

**"AI理解代码，人类理解业务"**

- AI负责代码分析和模式识别
- 人类提供业务上下文和判断
- 结合两者优势发现深层问题

### 对比传统工具

| 维度 | 传统工具 | Joern版 | AI版 |
|-----|---------|---------|------|
| 依赖 | 规则库 | Joern工具 | 无外部依赖 |
| 安装 | 简单 | 复杂 | 极简 |
| 理解 | 模式匹配 | 污点追踪 | 深度理解 |
| 误报 | 高 | 中 | 低 |
| 业务逻辑 | 弱 | 中 | 强 |
| 扩展性 | 需更新规则 | 需写查询 | AI自主分析 |

### 为什么这样设计

1. **降低使用门槛**
   - 无需安装复杂工具
   - 无需学习Joern语法
   - 开箱即用

2. **提高分析质量**
   - AI理解代码逻辑
   - 减少误报
   - 发现深层问题

3. **适应性强**
   - 适应各种项目结构
   - 适应各种编码风格
   - 适应各种业务场景

## 🔍 AI分析方法

### 方法1：关键词搜索 + 上下文分析

**适用**：注入类漏洞、敏感信息泄露

**流程**：
```
grep搜索 → 读取上下文 → AI分析 → 判断风险
```

### 方法2：业务流程追踪

**适用**：业务逻辑漏洞、认证授权

**流程**：
```
识别业务 → 读取完整流程 → AI分析数据流 → 识别缺陷
```

### 方法3：模式匹配 + 验证

**适用**：配置安全、密码存储

**流程**：
```
搜索模式 → 读取上下文 → AI判断用途 → 评估风险
```

## 总结

**核心理念**：
- AI理解 > 工具扫描
- 上下文分析 > 关键词匹配
- 业务逻辑 > 代码模式
- 端侧运行 > 云端分析

**设计目标**：
- 让AI像人类安全专家一样分析代码
- 无需复杂工具，开箱即用
- 端侧AI分析，深度理解代码逻辑
- 发现深层问题，减少误报
