# API安全最佳实践

> 通用的API安全知识

## 认证方式

### 1. API Key
**适用场景**：服务端到服务端  
**优点**：简单  
**缺点**：密钥泄露风险高

### 2. JWT
**适用场景**：前后端分离  
**优点**：无状态、可扩展  
**缺点**：无法主动失效

### 3. OAuth 2.0
**适用场景**：第三方授权  
**优点**：标准化、安全  
**缺点**：实现复杂

## 安全机制

### 1. HTTPS
**必须**：所有API必须使用HTTPS

### 2. 请求签名
**原理**：防止请求被篡改

**流程**：
1. 客户端：将请求参数排序后拼接
2. 客户端：使用密钥计算签名
3. 客户端：将签名放入请求头
4. 服务端：用相同方式计算签名
5. 服务端：对比签名是否一致

### 3. 时间戳
**原理**：防止重放攻击

**实现**：
- 请求带上时间戳
- 服务端检查时间戳是否在5分钟内
- 超时则拒绝

### 4. Nonce
**原理**：防止重放攻击

**实现**：
- 客户端生成随机字符串
- 服务端记录已使用的Nonce
- 重复的Nonce拒绝

### 5. 限流
**目的**：防止滥用

**维度**：
- IP级限流
- 用户级限流
- 接口级限流

## 输入验证

### 1. 参数验证
- 类型验证
- 长度验证
- 格式验证
- 范围验证

### 2. 白名单
- 字段名白名单
- 枚举值白名单
- 文件类型白名单

### 3. 防止注入
- SQL注入
- 命令注入
- XSS注入

## 输出安全

### 1. 敏感信息脱敏
- 手机号：138****5678
- 身份证：110***********1234
- 银行卡：6222 **** **** 1234

### 2. 错误信息
- 不返回详细错误
- 不暴露技术栈
- 不泄露路径信息

### 3. 响应头
- X-Frame-Options
- X-Content-Type-Options
- Content-Security-Policy

## 日志审计

### 1. 记录内容
- 请求时间
- 请求IP
- 用户ID
- 接口路径
- 请求参数（脱敏）
- 响应状态
- 响应时间

### 2. 不记录
- 密码
- Token
- 完整手机号
- 完整身份证

## API设计原则

### 1. RESTful规范
- GET：查询
- POST：创建
- PUT：更新
- DELETE：删除

### 2. 版本控制
- URL版本：/api/v1/users
- Header版本：Accept: application/vnd.api+json;version=1

### 3. 幂等性
- GET、PUT、DELETE必须幂等
- POST可以不幂等，但建议实现

### 4. 分页
- 限制单页最大数量
- 防止查询全表

## 常见漏洞

### 1. 未授权访问
**问题**：接口未验证权限  
**防护**：所有接口都要认证授权

### 2. 越权访问
**问题**：可以访问他人资源  
**防护**：验证资源所有权

### 3. 批量操作
**问题**：可以批量获取数据  
**防护**：限制批量操作数量

### 4. 信息泄露
**问题**：返回过多信息  
**防护**：只返回必要字段

### 5. 暴力破解
**问题**：可以暴力尝试  
**防护**：限流 + 验证码
