# 支付系统安全

> 支付系统的特殊安全要求

## 核心安全原则

### 1. 金额不可篡改
**原则**：永远不信任前端传来的金额

**实现**：
- 后端重新计算所有金额
- 对比前后端金额，不一致则拒绝
- 金额使用 Decimal 类型，避免浮点数精度问题

### 2. 幂等性保护
**原则**：同一笔支付只能成功一次

**实现**：
- 订单号全局唯一
- 支付前检查订单状态
- 使用分布式锁防止并发
- 幂等Token机制

### 3. 状态一致性
**原则**：订单状态与支付状态必须一致

**实现**：
- 使用事务保证原子性
- 异步回调要有重试机制
- 定时对账检查不一致

### 4. 资金安全
**原则**：防止资金损失

**实现**：
- 提现有限额和审核
- 异常交易实时告警
- 大额操作人工审核
- 完整的审计日志

## 常见攻击场景

### 1. 金额篡改
**攻击**：修改支付金额为0.01元  
**防护**：后端重新计算，拒绝不一致的请求

### 2. 重复支付
**攻击**：并发提交同一订单  
**防护**：订单状态检查 + 分布式锁

### 3. 回调伪造
**攻击**：伪造支付成功回调  
**防护**：验证回调签名 + IP白名单

### 4. 退款套利
**攻击**：利用退款规则漏洞获利  
**防护**：退款次数限制 + 异常检测

### 5. 订单篡改
**攻击**：修改订单商品或数量  
**防护**：订单创建后不可修改关键字段

## 必须实现的功能

### 支付流程
1. 创建订单（生成唯一订单号）
2. 发起支付（记录支付流水）
3. 接收回调（验证签名）
4. 更新状态（事务保证）
5. 通知业务（异步解耦）

### 退款流程
1. 验证退款条件（时间、次数、金额）
2. 创建退款单
3. 调用支付渠道退款
4. 接收退款回调
5. 更新订单状态

### 对账流程
1. 定时拉取支付渠道账单
2. 与本地订单对比
3. 标记差异订单
4. 人工处理差异

## 关键字段

### 订单表
- order_no：订单号（唯一）
- amount：订单金额（不可修改）
- status：订单状态
- pay_time：支付时间
- created_at：创建时间

### 支付流水表
- trade_no：支付流水号（唯一）
- order_no：关联订单号
- amount：支付金额
- channel：支付渠道
- status：支付状态
- callback_time：回调时间

## 安全检查清单

### 必查项（Critical）
- [ ] 金额由后端计算
- [ ] 订单号全局唯一
- [ ] 支付回调验证签名
- [ ] 订单状态流转正确
- [ ] 使用事务保证一致性

### 建议检查（High）
- [ ] 幂等性保护
- [ ] 重复支付检测
- [ ] 退款次数限制
- [ ] 异常交易告警
- [ ] 完整的审计日志

### 扩展检查（Medium）
- [ ] 定时对账
- [ ] 大额人工审核
- [ ] 支付超时处理
- [ ] 渠道容错机制

## 合规要求

### PCI DSS（支付卡行业数据安全标准）
- 禁止存储完整卡号
- 禁止存储CVV
- 传输必须加密
- 定期安全审计

### 反洗钱
- 大额交易报告
- 可疑交易监控
- 用户身份验证

### 数据保护
- 敏感数据加密存储
- 访问日志记录
- 数据备份和恢复
